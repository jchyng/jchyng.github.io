# Weiver — 뮤지컬 커뮤니티 플랫폼

뮤지컬 공연 정보, 배우·캐스팅 데이터를 제공하고 사용자들이 소통하는 **웹 커뮤니티 플랫폼**입니다.

프로젝트의 핵심 과제는 7만 건 이상의 뮤지컬 데이터를 안정적이고 빠르게 수집·처리하는 **크롤링 서버의 성능 최적화**였습니다. 단순 구현에 그치지 않고 JProfiler를 통한 메모리 분석까지 수행하며 **92%의 성능 개선**을 주도했습니다.

## 구현 내용

### 1. 크롤링 성능 92% 개선 — 병목 분석부터 최적화까지

초기 구현의 크롤링 소요 시간은 **약 2시간**으로, 개발과 테스트 사이클 전체에 심각한 병목이었습니다.

**병목 원인 파악**

| 원인 | 문제 |
|------|------|
| 동기 순차 처리 | 페이지 하나 처리 후 다음 페이지 요청 → 대기 시간 낭비 |
| 건별 INSERT | 7만 건 × 개별 쿼리 = DB 왕복 비용 누적 |

**개선 1단계: 배치(Bulk) INSERT 도입**

개별 INSERT 대신 메모리에 데이터를 모아 일괄 삽입 → **30분 단축**

> **난관**: 메모리에 대량 데이터를 누적하자 OOM(Out of Memory) 발생
> **해결**: JProfiler로 메모리 사용량을 실측한 뒤, JVM 튜닝 대신 안정적으로 처리 가능한 **1,000건 단위 분할 삽입** 적용

```java
// 배치 INSERT 핵심 로직
private static final int BATCH_SIZE = 1_000; // JProfiler 분석 결과 기반

public void processBatchInsert(List<MusicalData> dataList) {
    List<MusicalData> batch = new ArrayList<>();

    for (MusicalData data : dataList) {
        batch.add(data);
        if (batch.size() >= BATCH_SIZE) {
            dbRepository.batchInsert(batch);
            batch.clear();
        }
    }
    if (!batch.isEmpty()) {
        dbRepository.batchInsert(batch); // 나머지 처리
    }
}
```

**개선 2단계: 멀티스레드 병렬 처리**

뮤지컬 카테고리별 크롤링을 멀티스레드로 동시 실행.

> **최적화**: 스레드 수별 성능 벤치마크를 직접 수행하여 **5개 스레드**일 때 컨텍스트 스위칭 오버헤드 없이 최고 속도를 확인

### 2. Git Flow 도입 및 팀원 교육

초반에 팀원들이 브랜치 전략 없이 main에 직접 Push하는 문제가 있었습니다.

- Git Flow(feature / develop / main) 컨벤션을 문서화하고 팀 전체에 직접 교육
- PR 기반 코드 리뷰 프로세스 정착 → 충돌 사고 제로, 배포 안정성 향상

## 회고

단순 기능 구현을 넘어 **실제 병목을 데이터로 찾고 해결하는 과정**을 경험한 프로젝트입니다. JProfiler를 통한 정량적 분석과 가설-검증 기반의 단계적 최적화는 '느낌'이 아닌 **수치로 문제를 해결**하는 습관을 만들어줬습니다. 또한, Git 컨벤션을 팀 전체에 정착시키며 기술적 기여 외에도 **팀의 작업 방식을 개선하는 역할**을 수행할 수 있다는 것을 배웠습니다.
